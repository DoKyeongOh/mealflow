## 2. 배경 및 문서 간 정합

### 2.1 두 문서의 역할

- **`phase1_goal.md`**: 페이즈 1의 **포함·제외·완료 기준**을 명시한 **구현 범위의 1차 기준**이다.
- **`grocery_service_idea_summary.md`**: 서비스 전체 비전, **도메인 구조(종류 / 보관 상태 / 이력)**, 입력 부담 완화 방향, 장기 로드맵을 담은 **맥락 문서**이다.

### 2.2 범위 우선순위

페이즈 1에서 **기능 포함 여부·제외 여부에 대한 최종 판단은 `phase1_goal.md`를 따른다.**

아이디어 요약 문서의 「7. MVP 우선순위 제안」1차 항목에는 *보유 재료 기반 간단 레시피 추천*이 포함되어 있으나, `phase1_goal.md`의 **제외 범위**에 레시피 추천이 명시되어 있다. 본 페이즈 1 명세에서는 **레시피 추천은 페이즈 1 범위에 넣지 않는다.** 향후 단계에서 아이디어 요약과 재정렬할 수 있다.

---

## 3. 페이즈 1 목표

### 3.1 사업·제품 목표

회원 관리와 기본 식료품 관리를 중심으로, 사용자가 **자신의 식료품 상태**와 **변경 이력**을 기록할 수 있는 **최소 기능**을 구축한다.

화려한 기능보다 **식료품 상태 데이터와 이력 데이터가 안정적으로 관리되는 구조**를 만드는 것이 핵심이다.

### 3.2 한 줄 정의

> **식료품 상태와 이력을 관리할 수 있는 최소 기능의 기반 시스템 구축 단계**

### 3.3 아이디어 요약과의 연결 (비범위 가치)

장기적으로 서비스는 「집에 있는 재료로 무엇을 먹을지 빠르게 정하게 해주는 것」「입력 마찰 최소화」를 지향한다. 페이즈 1에서는 그 **토대가 되는 데이터(종류·현재 보관·이력)**만 확보하고, 레시피·장보기·고도화된 알림 등은 다음 단계로 미룬다.

---

## 4. 범위

### 4.1 포함 범위

| 영역 | 요약 |
|------|------|
| 회원 | 가입, 로그인, 로그아웃, **사용자별 데이터 분리** |
| 식료품 종류 | CRUD, 이름, 기본 보관 장소, 평균 유통기한 기본값 |
| 보관 상태 | 현재 보유 항목 CRUD, 수량·단위·유통기한·보관 장소, 목록 조회 |
| 변경 이력 | 입고·사용·폐기·수정, 항목별·전체 조회 |
| 편의 | 평균 유통기한 자동 입력, 기본 보관 장소 자동 선택, 유통기한 임박 표시, 최근 등록 항목 간단 재사용 |

### 4.2 제외 범위 (페이즈 1에서 구현하지 않음)

- OCR·영수증 기반 일괄 등록
- 복잡한 UX 최적화 (단, 페이즈 1에 명시된 편의 기능은 포함)
- 레시피 추천
- 장보기 추천
- 식단·플랜 기능

## 5. 사용자 및 권한

| 구분 | 설명 |
|------|------|
| 비회원 | 식료품 데이터에 접근하지 않는다 (회원 기능 전제는 구현 정책에 따름). |
| 회원 | 본인 계정에 연결된 식료품 종류·보관 상태·이력만 조회·수정할 수 있다. |
| 데이터 분리 | 모든 식료품 관련 엔티티는 사용자(계정) 단위로 격리된다. |

---

## 6. 기능 요구사항

### 6.1 회원 관리

| ID | 요구사항 | 상세 |
|----|----------|------|
| M-01 | 회원 가입 | 계정을 생성할 수 있다. |
| M-02 | 로그인 | 등록된 자격으로 세션(또는 토큰) 기반 인증을 통해 접근할 수 있다. |
| M-03 | 로그아웃 | 인증 상태를 종료할 수 있다. |
| M-04 | 데이터 분리 | 다른 사용자의 식료품 종류·보관 상태·이력에 접근할 수 없다. |

**비고:** 인증 방식(이메일·소셜 등)과 세부 검증 규칙은 별도 보안·정책 문서 또는 구현 단계에서 정할 수 있으나, 위 요구사항을 만족해야 한다.

---

### 6.2 식료품 종류 관리

식료품의 **마스터(기준) 정보**를 관리한다. 아이디어 요약의 「식료품 종류」에 대응한다.

| ID | 요구사항 | 상세 |
|----|----------|------|
| T-01 | 등록 | 새 식료품 종류를 추가할 수 있다. |
| T-02 | 수정 | 기존 종류의 속성을 변경할 수 있다. |
| T-03 | 삭제 | 종류를 삭제할 수 있다. 삭제 시 보관 중인 항목·이력 처리 규칙은 구현 시 정의(차단·연쇄 삭제·소프트 삭제 등). |
| T-04 | 이름 | 종류를 식별하는 이름을 둔다. |
| T-05 | 기본 보관 장소 | 신규 보관 항목 등록 시 참고할 **기본 보관 장소**를 종류 단위로 둔다. |
| T-06 | 평균 유통기한 기본값 | 일수 등으로 저장하며, 보관 항목 등록 시 유통기한 자동 제안에 사용한다. |

**참고:** 기본 보관 장소 예시로 냉장고·냉동고가 언급되어 있으나, 사용자 정의 값 허용 여부는 제품 정책에 따르되, 최소한 위 예시를 선택지로 쓸 수 있어야 한다.

---

### 6.3 식료품 보관 상태 관리

**현재 집에 보유 중인** 식료품의 상태를 관리한다. 상태성 데이터이며, 아이디어 요약의 「현재 상태 테이블」에 해당한다.

| ID | 요구사항 | 상세 |
|----|----------|------|
| S-01 | 등록 | 특정 식료품 종류를 기준으로 보유 항목을 추가할 수 있다. |
| S-02 | 수정 | 수량·단위·유통기한·보관 장소 등 허용 필드를 변경할 수 있다. |
| S-03 | 삭제 | 보유 항목을 목록에서 제거할 수 있다. |
| S-04 | 수량·단위 | 수량과 단위를 입력·표시한다. |
| S-05 | 유통기한 | 유통기한을 입력·표시한다. |
| S-06 | 보관 장소 | 보관 장소를 입력·표시한다. |
| S-07 | 목록 조회 | 현재 보유 식료품 목록을 조회할 수 있다. |

**도메인 메모 (아이디어 요약):** 장기적으로 「상태」 필드를 두어 관리 단계를 표현할 수 있으나, 페이즈 1 목표 문서에 명시되지 않았다면 **필수 항목으로 두지 않고**, 필요 시 확장 필드로 설계할 수 있다.

---

### 6.4 식료품 변경 이력 관리

상태 변화는 **이벤트 로그**로 누적한다. 아이디어 요약의 「이벤트 로그 테이블」에 해당한다.

| ID | 요구사항 | 상세 |
|----|----------|------|
| H-01 | 입고 이력 | 입고 행위를 기록할 수 있다. |
| H-02 | 사용 이력 | 사용 행위를 기록할 수 있다. |
| H-03 | 폐기 이력 | 폐기 행위를 기록할 수 있다. |
| H-04 | 수정 이력 | 보관 상태 또는 관련 정보의 수정을 기록할 수 있다. |
| H-05 | 항목별 조회 | 특정 보유 항목(또는 종류)에 대한 이력을 조회할 수 있다. |
| H-06 | 전체 조회 | 사용자 기준으로 이력을 조회할 수 있다(필터·페이징은 구현 수준에서 정의). |

**원칙:** 현재 상태는 별도로 관리하고, 변화는 이력으로 남긴다. **상태 변경 시 이력이 자동으로 쌓이는 흐름**이 페이즈 1 완료 기준과 검증 포인트의 핵심이다.

---

### 6.5 기본 편의 기능

입력 부담을 줄이기 위한 **최소 편의**만 포함한다 (아이디어 요약 4장 방향과 정합).

| ID | 요구사항 | 상세 |
|----|----------|------|
| C-01 | 평균 유통기한 자동 입력 | 식료품 종류의 평균 유통기한 기본값을 이용해, 등록 시 유통기한 필드를 자동 채우거나 제안한다. |
| C-02 | 기본 보관 장소 자동 선택 | 종류에 설정된 기본 보관 장소를 신규 등록 시 기본값으로 제안한다. |
| C-03 | 유통기한 임박 표시 | 기준일 대비 임박한 항목을 사용자가 식별할 수 있게 표시한다(임계일·배지 등 구체 UI는 설계 시 정의). |
| C-04 | 최근 등록 항목 간단 재사용 | 최근에 등록한 항목을 빠르게 다시 쓸 수 있는 수단을 제공한다(예: 최근 목록에서 선택하여 재등록). |

**제외:** 영수증 OCR, 복잡한 원터치 UI 전면 적용은 페이즈 1 범위 밖이다. 다만 조작 수를 줄이는 방향은 C-01~C-04에서 일부 반영한다.

## 7. 데이터 구조 개요 (페이즈 1)

다음 세 축을 명확히 가져간다.

| 개념 | 역할 |
|------|------|
| 식료품 종류 | 마스터 정보(이름, 기본 보관, 평균 유통기한 기본값) |
| 식료품 보관 상태 | 현재 보유량·유통기한·보관 장소 등 **현재 스냅샷** |
| 식료품 변경 이력 | 입고·사용·폐기·수정 등 **시간순 이벤트 누적** |

이 구조는 이후 UX 개선·추천 기능의 기반이 된다.

## 8. 핵심 검증 포인트 (기능 관점)

페이즈 1 목표 문서의 검증 포인트를 기능 명세에 대응시키면 다음과 같다.

1. 사용자가 식료품을 **직접 등록하고 관리**할 수 있는가.
2. **현재 상태**와 **이력**이 자연스럽게 이어지는가.
3. **최소 기능만으로** 서비스의 기본 흐름(등록 → 조회 → 변경·이력)이 성립하는가.

---

## 9. 완료 기준 (인수 조건)

다음을 모두 만족하면 페이즈 1 기능 관점의 완료로 본다.

1. 회원이 로그인 후 **자신의 식료품**만 관리할 수 있다.
2. 식료품 **현재 상태**를 조회할 수 있다.
3. 변경 시 **이력이 자동 기록**된다(정책에 따른 예외가 있다면 문서화).
4. **유통기한 임박** 항목을 확인할 수 있다.
