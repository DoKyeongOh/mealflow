# UI 셸·레이아웃 참고

## 1. 문서 역할 (와이어프레임과의 관계)

| 구분 | 문서 | 역할 |
|------|------|------|
| **규범** | `docs/ui/phase1_screen_specification.md`(본 문서에서 「와이어프레임」) | 화면 ID, 정보 구조, 페이즈 1에 **반드시 맞춰야 할** 배치 원칙(중앙 컨테이너 등) |
| **참고** | 본 문서 (`ui_shell_reference.md`) | 데스크톱·모바일 **접속감을 맞추기 위한 앱 셸 패턴**, 외부 서비스 UI를 **영감·설명** 수준으로 적어 둔다. 픽셀 단위 스펙이나 브랜드 복제가 아니다. |

**왜 따로 두는가**

- 화면 정의서에 스크린샷·“OO 같이” 설명을 길게 넣으면, **기능 요구와 감성 레퍼런스**가 한 파일에 섞여 업데이트 단위가 달라진다.
- 참고만 바꿀 때는 본 문서만 고치면 되고, **구현 우선순위·완료 판단**은 계속 와이어프레임(`docs/ui/phase1_screen_specification.md`)·기능 명세에 둔다.

## 2. 참고 패턴: 상단 바 + 좌측 사이드바 + 중앙 본문

**Google 항공편(Flights)** 류 데스크톱 UI를 떠올리면 된다.

| 영역 | 역할 |
|------|------|
| **상단 바** | 서비스 로고, 주요 탭/섹션(선택), 설정·프로필 등 유틸. 전역에서 유지되면 “한 앱 안” 느낌이 난다. |
| **좌측 사이드바** | 보조 내비게이션(메뉴 항목, 설정 링크 등). 데스크톱에서는 **항상 보이거나** 접었다 펼 수 있게 둘 수 있다. |
| **햄버거 버튼** | 사이드바를 **토글**하는 진입점. 상단 좌측에 두는 패턴이 흔하다. |
| **중앙 본문** | 실제 화면(목록·폼 등). **가운데 정렬·최대 폭 제한**을 두면 네이버 메인형 “중앙 덩어리”와도 잘 맞는다. |

**모바일과의 연결**

- 같은 정보 구조를 유지한 채, 좌측 사이드바만 **오버레이 드로어**(햄버거 탭 시 슬라이드)로 바꾸면, 데스크톱에서 익힌 메뉴 위치·이름이 모바일에서도 이어져 **접속감**이 비슷해지기 쉽다.
- 상단 바는 좁은 폭에서도 남기고, 본문은 세로 스택으로 두면 CSS·컴포넌트 수정 범위를 줄일 수 있다(완전히 다른 모바일 IA를 새로 짓는 것보다).

## 3. 기존 문서와의 정합

- `wireframe.md` §2의 **중앙 컨테이너·여백**과 모순되지 않는다. 본 패턴은 그 위에 **내비게이션 뼈대(셸)** 를 얹는다고 보면 된다.
- **2열(왼쪽 주 콘텐츠 + 오른쪽 보조 위젯)** 은 INV-01 등에서 여전히 쓸 수 있다. “왼쪽 = 사이드바(내비)”와 “왼쪽 = 주 카드 열”은 **역할이 다르므로** 한 화면에서 사이드바 + 중앙 2열(주+우)을 함께 쓰는 설계도 가능하다(시각 와이어에서 밀도 조절).

---

## 4. 본 문서에서 다루지 않는 것

- 색·폰트·다크 모드 여부(디자인 시스템 또는 별도 시안).
- 컴포넌트 라이브러리 선택.
- 접근성·포커스 트랩 등 세부 인터랙션 스펙(구현 시 보완).
