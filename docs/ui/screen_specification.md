## 2. 레이아웃·배치 원칙 (가운데 중심)

웹 기준 **본문 콘텐츠는 뷰포트 가운데에 고정 폭(또는 최대 폭)으로 모아** 배치한다. 참고로 **네이버(naver.com) 메인**과 같이, 좌우에 여백이 생기고 중앙에 주요 정보가 모이는 **센터 정렬·컨테이너형** 레이아웃을 기본으로 한다.

| 원칙 | 설명 |
|------|------|
| **중앙 컨테이너** | 주 콘텐츠는 `max-width`가 있는 하나의 블록 안에 두고, 가로로 **화면 중앙에 정렬**한다. 창이 넓어져도 본문이 좌우 끝까지 늘어나 전체를 채우지 않게 한다(읽기·스캔 부담 완화). |
| **좌우 여백** | 컨테이너 바깥은 배경색 또는 단색 여백으로 두어, **포털형**처럼 “가운데 한 덩어리”가 보이게 한다. |
| **앱 셸·내비(권장)** | **상단 바 + 햄버거로 여는 좌측 사이드바 + 중앙 본문** 조합을 쓰면, 모바일에서 사이드바를 **드로어**로 바꿀 때도 메뉴 구조가 이어져 **접속감**을 맞추기 쉽다. 구체 예·참고 UI는 **`docs/phase1/ui_shell_reference.md`** (본 문서와 달리 **참고 전용**). |
| **2열 구조(콘텐츠 내)** | 본문 **안에서** 네이버 메인처럼 **왼쪽 넓은 열(주 콘텐츠)** + **오른쪽 좁은 열(보조 위젯)** 을 쓸 수 있다(예: INV-01에서 목록+임박 요약). **좌측 사이드바(전역 내비)** 와는 별개이며, 화면마다 오른쪽 열을 생략하거나 단일 열만 쓸 수 있다. |
| **구역 구획** | 카드·박스 단위로 영역을 나누고(테두리·배경 대비), 섹션 제목과 본문 위계를 명확히 한다. |
| **반응형** | 좁은 뷰포트에서는 콘텐츠 2열을 **1열로 수직 스택**하고, 동일하게 “중앙 기준으로 콘텐츠 폭 제한 + 좌우 패딩”을 유지한다. 사이드바는 오버레이로 전환하는 방식이 일반적이다. |

**주의:** 레이아웃 **패턴**만 참고한다. 색·로고·서비스 아이덴티티는 외부 사이트를 복제하지 않는다.

## 3. 페이즈 1 범위 정렬

- 기능적 최종 기준은 **`phase1_functional_specification.md`**이다.
- 아이디어 요약의 레시피·장보기·식단 등은 **페이즈 1 화면으로 두지 않는다**.
- 인증은 **소셜 로그인(IdP·OAuth 연동)**을 기본으로 한다. 이메일·비밀번호 등 로컬 계정은 정책에 따라 선택적으로 둘 수 있다. **로그인·회원가입·로그아웃**에 대응하는 화면·흐름은 본 문서에 포함한다.

---

## 4. 정보 구조 가정

- 로그인 후 **허브(주 진입점)**는 **현재 보유 식료품 목록**으로 둔다. 사용자가 가장 자주 보는 화면이며, 완료 기준(상태 조회·임박 확인)과도 맞는다.
- **식료품 종류**는 마스터 데이터 관리 화면으로 분리한다(목록에서 보관 항목 등록 시 선택).
- **변경 이력**은 **전체 타임라인**과 **특정 보관 항목 상세의 이력** 두 축으로 나눈다.

---

## 5. 화면과 “모달/드로어”

- 동일한 “페이지”로 할지, **오버레이(모달·바텀시트·사이드 패널)**로 할지는 구현·시각 와이어 단계에서 정한다.
- 본 문서에서는 **사용자가 인지하는 화면 단위(목적 단위)**로만 정의하며, UI 패턴은 “후보”로만 적는다.

## 7. 구역별 화면 정의

### 7.1 인증 (AUTH)

#### AUTH-01 로그인

| 항목 | 내용 |
|------|------|
| **목적** | 등록된 자격으로 서비스에 진입한다 (M-02). |
| **진입** | 비로그인 사용자가 앱/웹 최초 접근 시, 또는 세션 만료 후. 가입·연동 완료 후 이 화면 또는 자동 로그인 후 허브로 이동(정책에 따름). |
| **주요 구성(콘셉트)** | **기본: 소셜 로그인.** 제공자별 버튼(예: Google 등), 클릭 시 IdP 로그인 화면으로 이동·콜백 후 세션 수립. 실패·취소 시 안내 메시지. 회원가입(또는 “처음 오셨나요?”)으로 이어지는 링크. **선택:** 이메일·비밀번호 필드와 로그인 제출은 로컬 계정을 둘 때만 보조적으로 배치한다. **배치:** 중앙 컬럼에 **소셜 버튼을 시각적 중심**으로 한 카드(2열 사이드바 없음). |
| **주요 액션** | 소셜 제공자 선택 후 IdP에서 인증 완료, (선택) 이메일 로그인 제출, 회원가입·안내 화면 이동. |
| **연결 요구사항** | M-02, M-04(성공 후 본인 데이터만). |
| **비고** | 기본 UX는 소셜 중심. 이메일 로그인을 넣을 때만 “비밀번호 찾기”·AUTH-03 등을 검토한다. |

#### AUTH-02 회원가입

| 항목 | 내용 |
|------|------|
| **목적** | 신규 계정을 만든다 (M-01). |
| **진입** | 로그인 화면에서 회원가입 링크, 랜딩 CTA 등. |
| **주요 구성(콘셉트)** | **기본: 소셜로 가입.** AUTH-01과 동일한 IdP 버튼으로 “가입하기”; 최초 인증 시 서버에서 계정을 생성하는 흐름을 기본으로 한다. 법적 요건에 따라 **약관·개인정보 동의**는 IdP 이동 전에 체크하거나, 콜백 직후 별도 단계로 받는다. (선택) 이메일·비밀번호 가입 폼. 로그인 화면으로 돌아가기. **배치:** AUTH-01과 동일하게 **중앙 정렬 단일 컬럼**, 소셜 버튼이 주 시선이 되도록 한다. |
| **주요 액션** | 소셜 선택 후 약관 동의(정책 순서에 따름) → IdP 인증 → (필요 시) 프로필 보완, (선택) 이메일 가입 제출, 로그인 화면 이동. |
| **연결 요구사항** | M-01. |
| **비고** | 소셜만 쓰면 로그인·가입 화면이 버튼 구성이 거의 같고, 문구·약관 노출 순서만 다를 수 있다. 필드·검증·토큰 처리는 보안·정책 문서 또는 구현 시 확정. |

### 7.2 보관 상태 (INV)

#### INV-01 보유 식료품 목록 (허브)

| 항목 | 내용 |
|------|------|
| **목적** | 현재 보유 중인 식료품을 한눈에 보고, 등록·상세·임박 파악의 시작점이 된다 (S-07, 완료 기준 2·4). |
| **진입** | 로그인 직후 기본 랜딩. 하단/상단 네비에서 “목록” 선택 시. |
| **주요 구성(콘셉트)** | 보유 항목 리스트(종류명, 수량·단위, 유통기한, 보관 장소 등 요약), **유통기한 임박 시각적 구분** (C-03), 정렬·필터(최소: 임박 우선 등—구현 수준에서 정의), 빈 상태 안내, 신규 등록 진입점, **최근 등록 항목 재사용** 진입 (C-04), 다른 구역(종류 관리, 전체 이력, 계정)으로 가는 내비게이션. **배치:** §2의 **2열**을 적용하기 좋은 화면이다. 예: 왼쪽에 목록·필터, 오른쪽에 **임박 요약·최근 재사용·계정/로그아웃** 등 보조 블록. |
| **주요 액션** | 항목 탭 → 상세, 신규 등록, 최근 항목 재등록, (선택) 항목에서 빠른 사용/폐기 등은 INV-03과 역할 분담. |
| **연결 요구사항** | S-07, C-03, C-04, M-04. |
| **비고** | 레시피·장보기 배너는 페이즈 1 범위 밖. |

#### INV-02 보유 식료품 등록

| 항목 | 내용 |
|------|------|
| **목적** | 특정 식료품 종류를 기준으로 보유 항목을 추가한다 (S-01). |
| **진입** | 목록의 “추가”, 빈 상태의 CTA, (C-04) 최근 항목에서 “같은 형태로 다시 등록” 등. |
| **주요 구성(콘셉트)** | 식료품 종류 선택(검색·최근·신규 종류 만들기 링크), 수량·단위, 유통기한( **종류의 평균 유통기한으로 자동 제안** (C-01)), 보관 장소( **종류의 기본 보관 장소 자동 선택** (C-02)), 제출·취소. **배치:** 주 폼은 중앙 컬럼; 보조 설명은 같은 열 또는 오른쪽 좁은 열에 둘 수 있다. |
| **주요 액션** | 저장, 취소, 필요 시 “새 종류 등록”으로 TYP-02 연결. |
| **연결 요구사항** | S-01, S-04~S-06, C-01, C-02, H-01(저장 시 입고 이력 정책에 따라). |
| **비고** | 등록과 동시에 이력이 어떻게 쌓이는지는 백엔드·정책과 일치해야 함(기능 명세 6.4, 9장). |

#### INV-03 보유 식료품 상세·편집

| 항목 | 내용 |
|------|------|
| **목적** | 한 보유 항목의 상세를 보고, 수정·삭제하거나 사용/폐기 등 이력과 연결되는 조작을 한다 (S-02, S-03, H-02, H-03, H-04, H-05). |
| **진입** | 목록에서 항목 선택. |
| **주요 구성(콘셉트)** | 현재 스냅샷 표시(수량, 단위, 유통기한, 보관 장소, 연결된 종류명), 편집 폼, 삭제, **해당 항목의 변경 이력** 구역 또는 “이력 전체 보기”로의 링크(H-05), 사용·폐기 등 빠른 액션(있을 경우). **배치:** 상세는 왼쪽 주 열, 이력 타임라인은 아래 또는 오른쪽 열로 나누는 방식을 허용한다. |
| **주요 액션** | 수정 저장, 삭제, 사용 기록, 폐기 기록, (선택) 수정으로 인한 이력(H-04). |
| **연결 요구사항** | S-02, S-03, H-02~H-05, 일관성·트랜잭션(비기능). |
| **비고** | “상태(관리 단계)” 필드는 페이즈 1 필수 아님(기능 명서 6.3). |

### 7.3 식료품 종류 마스터 (TYP)

#### TYP-01 식료품 종류 목록

| 항목 | 내용 |
|------|------|
| **목적** | 등록된 식료품 종류 마스터를 조회하고, 추가·수정·삭제의 진입점을 제공한다 (T-01~T-03). |
| **진입** | 글로벌 내비게이션 “종류 관리”, 보관 등록 흐름에서 “새 종류 만들기” 등. |
| **주요 구성(콘셉트)** | 종류별 행(이름, 기본 보관 장소, 평균 유통기한 요약), 검색(선택), 신규 등록, 행 선택 → 편집. **배치:** 테이블·리스트는 중앙 컨테이너 안에서 전체 폭 사용. |
| **주요 액션** | 항목 추가(TYP-02), 항목 편집·삭제(T-03 처리 규칙은 구현 시 명시). |
| **연결 요구사항** | T-01~T-04. |
| **비고** | 삭제 시 보유 항목이 있을 때 차단/연쇄/소프트 삭제 중 무엇을 쓸지 정책 문서 또는 기술 명세와 맞출 것. |

#### TYP-02 식료품 종류 등록·편집

| 항목 | 내용 |
|------|------|
| **목적** | 종류를 새로 만들거나 속성을 바꾼다 (T-01, T-02, T-05, T-06). |
| **진입** | 종류 목록의 추가/특정 행 편집. |
| **주요 구성(콘셉트)** | 이름, 기본 보관 장소(최소 예시: 냉장고·냉동고·사용자 정의 허용 여부는 정책), 평균 유통기한 기본값(일수 등), 저장·취소. **배치:** 폼 중심 단일 열 또는 INV-02와 동일 패턴. |
| **주요 액션** | 저장, 취소. |
| **연결 요구사항** | T-01, T-02, T-04~T-06. |
| **비고** | INV-02와 필드 일관성 유지. |
